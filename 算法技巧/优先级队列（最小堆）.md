

在 Java 后端开发中，`PriorityQueue`（优先级队列）是一个非常重要的数据结构，尤其在处理“前 K 个高频元素”**、**“合并 K 个有序链表”**或**“任务调度”等场景时。

它与普通队列（FIFO，先进先出）最大的不同在于：**它不按插入顺序排序，而是按元素的优先级排序。**

---

### 1. 核心特性

- **无序插入，有序弹出**：你可以乱序把元素扔进去，但每次调用 `poll()` 或 `peek()` 时，拿到的永远是当前队列中“优先级最高”的那个元素。
    
- **底层实现**：它是基于**二叉堆**（Binary Heap）实现的，具体来说通常是一个**小顶堆**（Min-Heap）。
    
- **不保证整体有序**：如果你遍历 `PriorityQueue`，你会发现它的顺序看起来是乱的。它只保证**堆顶**（即第一个元素）是全局最小的。
    
- **线程不安全**：如果需要多线程环境，请使用 `PriorityBlockingQueue`。
    
- **不允许 NULL**：插入 `null` 会抛出 `NullPointerException`。
    

---

### 2. 底层原理：二叉堆

`PriorityQueue` 实际上是一棵**完全二叉树**，并存储在一个动态数组中。

- **小顶堆逻辑**：父节点的值总是小于或等于其子节点的值。
    
- **时间复杂度**：
    
    - **入队 (`add`/`offer`)**：$O(\log n)$ —— 插入末尾并向上调整（Sift Up）。
        
    - **出队 (`poll`)**：$O(\log n)$ —— 移除堆顶，将末尾补到堆顶并向下调整（Sift Down）。
        
    - **查看堆顶 (`peek`)**：$O(1)$ —— 直接返回数组第一个元素。
        

---

### 3. 如何定义优先级？

默认情况下，`PriorityQueue` 是**小顶堆**（数字越小优先级越高）。如果你想改变规则（比如做大顶堆），有两种方式：

#### A. 自然排序（Natural Ordering）

元素本身实现 `Comparable` 接口（例如 Integer, String 自带这个）。

#### B. 比较器排序（Comparator）—— **面试常用**

在构造函数中传入一个 `Comparator`。这在处理复杂对象（如链表节点或自定义类）时必不可少。

Java

```
// 创建一个大顶堆（从大到小排）
PriorityQueue<Integer> maxHeap = new PriorityQueue<>((a, b) -> b - a);

// 处理合并 K 个链表时的写法
PriorityQueue<ListNode> pq = new PriorityQueue<>((a, b) -> a.val - b.val);
```

---

### 4. 为什么合并 K 个链表要用它？

“合并 K 个链表”效率问题中

- **如果不使用分治，也不使用 PriorityQueue**：你可能需要每次遍历 $K$ 个链表的头节点来找最小值，复杂度是 $O(NK)$。
    
- **使用 PriorityQueue**：
    
    1. 把 $K$ 个链表的头节点全扔进堆里（堆大小为 $K$）。
        
    2. 每次从堆顶弹出最小节点，接到结果链表后。
        
    3. 如果被弹出的节点还有下一个节点，就把下一个节点再扔进堆。
        
- **效率提升**：因为堆的弹出和插入都是 $O(\log K)$，总共 $N$ 个节点，所以总复杂度是 **$O(N \log K)$**。
    

---

### 5. 总结与面试高频考点

- **考点 1：扩容机制**。`PriorityQueue` 是会自动扩容的，初始容量通常是 11。
    
- **考点 2：Top K 问题**。
    
    - 找最大的前 K 个数：用**小顶堆**（堆满时，如果新数比堆顶大，替换堆顶。最后堆里剩下的就是最大的 K 个）。
        
    - 找最小的前 K 个数：用**大顶堆**。
        
- **考点 3：与 `TreeSet` 的区别**。
    
    - `TreeSet` 是基于红黑树的，它保证**所有**元素始终有序，且不允许重复。
        
    - `PriorityQueue` 只保证**堆顶**有序，允许重复，且由于堆的结构更简单，其入队出队的常数开销通常比红黑树更小。
